#+title: Majutsu Gerrit Client
#+filetags: design gerrit

* Goal

Provide a complete, self-contained Gerrit client inside the Majutsu
repository, replacing the user's external dependency on
[[https://github.com/thisch/gerrit.el][gerrit.el]] and the
[[https://git.sr.ht/~niklaseklund/egerrit][egerrit]] package.

The client lives under the =majutsu-gerrit-= prefix and is built on
Majutsu's existing infrastructure modules.  Where features overlap, the
client picks the better-engineered design from either upstream and
re-implements it on top of Majutsu primitives instead of borrowing the
upstream code at runtime.

The current =majutsu-gerrit.el= already wraps =jj gerrit upload= and
related sub-commands through a transient.  The new modules add the
missing pieces: REST, dashboard, change/topic info views, inline diff
review, draft comments and a local read-state database.  The purpose is
not to clone Polygerrit, whose keyboard review UI is already strong, but
to make Gerrit review state part of the local jj/Emacs environment: local
working-copy jumps, jj upload/download loops, xref/LSP navigation,
searchable text, Org links and agent-readable state.  At the same time,
the existing =--with-gerrit-context= and =--with-egerrit-context= bridges
in =majutsu-gerrit.el= are removed, since the new in-tree implementation
makes them obsolete.

The new modules are designed to be extractable as a standalone package
in the future, so they only depend on Majutsu's =majutsu-process=,
=majutsu-base=, =majutsu-mode=, =majutsu-section=,
=majutsu-completion=, =majutsu-selection=, =majutsu-config= and
=majutsu-remote= modules, and not on jj-specific logic such as
=majutsu-jj.el= except in the existing =majutsu-gerrit.el= entry file.

* User Interface

This section documents the parts of the Gerrit client that are already
implemented.  Later sections describe the design for remaining modules.

** Gerrit Dispatch

~M-x majutsu-gerrit-dispatch~ opens the top-level Gerrit dispatcher from
any Gerrit buffer.  It is bound to ~?~ in Gerrit dashboard buffers and,
with Evil active, in normal state as well.

The dispatch is the intended entry point for all Gerrit-related
commands, including future change-detail and review buffers.  The
initial keys are:

| Key | Command                              | Description                            |
|-----+--------------------------------------+----------------------------------------|
| ~u~ | ~majutsu-gerrit-upload-transient~    | Open the =jj gerrit upload= transient. |
| ~R~ | ~majutsu-gerrit-dashboard-transient~ | Open the dashboard options transient.  |
| ~d~ | ~majutsu-gerrit-dashboard-download-change~ | Download the change at point.    |
| ~g~ | ~majutsu-gerrit-dashboard-refresh-buffer~  | Refresh the current Gerrit buffer. |
| ~q~ | ~transient-quit-one~                 | Close the dispatch.                    |

** Dashboard

~M-x majutsu-gerrit-dashboard~ opens a section-based Gerrit dashboard
for the current jj repository.  The dashboard is backed by a single
buffer-local state object (~majutsu-gerrit-dashboard--state~) that
contains a tree of /items/.  Each item is either a query or a group of
queries.  The state is the source of truth; =magit-section= is used only
for rendering and as a cursor into the state.

Top-level dashboard items are rendered as sections.  Query sections show
their title, change count, and the underlying Gerrit query string.  A
query section can be empty; it is hidden only when =--hide-empty= is
active or the query itself is marked hidden-empty.  Group sections
contain nested query sections and are shown only when at least one child
query has visible results.

*** Built-in dashboard items

The default dashboard uses the same Polygerrit-style queries as Gerrit's
own user dashboard:

- Has draft comments
- Your turn
- Work in progress
- Outgoing reviews
- Incoming reviews
- CCed on
- Recently closed

You can add arbitrary query sections with ~a~ and user dashboard groups
with ~U~.  A user dashboard group targets a specific Gerrit user
(default ~self~) and expands into the Polygerrit template with ~${user}~
replaced.  Self-only sections (such as "Has draft comments" and "Work in
progress") are omitted when the target is not ~self~.

*** Editing and removing sections

- ~e~ :: Edit the query string of the current query section, or change
the target user of the current user dashboard group.
- ~E~ :: Rename the title of the current query or group section.
- ~k~ :: Remove the current section.  Removing requires confirmation.

* Gerrit server facts used by the design

This section is based on the Gerrit source tree in =~/Projects/gerrit=,
not only on the two Emacs packages.

** REST authentication and JSON shape

- Authenticated REST calls are made by prefixing the endpoint with =/a/=
  (Gerrit documentation: =Documentation/rest-api.txt=,
  Authentication).  This uses HTTP Basic credentials from the user's
  account settings and does not require XSRF tokens.
- JSON responses are XSSI-protected by a leading magic line:
  =)]}'=.  The client must strip the whole line before JSON parsing.
- Tools should request compact JSON by either passing =pp=0= or sending
  =Accept: application/json=.  Majutsu does both for JSON endpoints.
- Error responses are usually plain text, not JSON.  The low-level REST
  function therefore must not assume that every response body is JSON.

** Change identity

Gerrit has multiple identifiers and they are not interchangeable:

- =ChangeInfo.id= is a REST resource id in encoded form.  In Gerrit 3.14
  it is =url(project)~number=; this is set in
  =java/com/google/gerrit/server/change/ChangeJson.java=.
- =ChangeInfo.triplet_id= is =url(project)~url(branch)~url(change key)=.
- =ChangeInfo._number= is the global numeric change number.
- =ChangeInfo.change_id= is Gerrit's stable change key.  For traditional
  Git clients this is the =I...= value from the commit-message
  =Change-Id= footer.  For Jujutsu-aware Gerrit, this may instead be the
  raw 32-character jj change id read from the Git commit-object
  =change-id= header, e.g. =mqywrmzsrppzuprkpuutrxsmplmkkkzt=.

Majutsu treats raw jj-style change keys as first-class Gerrit change IDs,
not as an implementation detail.  A jj-style key matches
=^[k-z]\{32\}$= and should be used directly for local jj revset lookup,
cleanup decisions and dashboard display.  The older converted key form
=I<forward-hex-jj-id>6a6a6964= is legacy compatibility only; new
header-based uploads should not create it.

For REST endpoints under =/changes/{change-id}=, prefer =_number= when
we already have a =ChangeInfo= object.  It avoids re-encoding
=ChangeInfo.id=, whose project component is already URL encoded by the
server.  Gerrit also accepts other identifiers, but
=Documentation/rest-api-changes.txt= explicitly says they require extra
lookups and may produce false positives/negatives.

For search queries (=GET /changes/?q=...=), use the user's query string,
a numeric change number, or a change key (=I...= or jj-style raw id) as a
query term.  Do not pass =ChangeInfo.id= to a search query and expect it
to work as a resource id; modern Gerrit treats query parsing and REST
resource parsing separately.  This distinction is the root cause of an
egerrit bug we already fixed.

All IDs that appear in URL path segments must be URL encoded.  This
includes project names, branch names, file paths, labels and comment IDs.
File paths must be encoded as a single path segment, so slashes become
=%2F=; magic paths such as =/COMMIT_MSG= and =/PATCHSET_LEVEL= must also
be encoded before being inserted into =/files/{file-id}= or comment
payloads.

** Change query behavior

=java/com/google/gerrit/server/restapi/change/QueryChanges.java= shows
that =GET /changes/= accepts repeated =q= parameters.  With one query,
Gerrit returns a list of =ChangeInfo= objects.  With more than one query,
Gerrit returns a list of lists in the same order as the repeated =q=
parameters.  The REST layer preserves the raw shape; higher-level
helpers may normalize it.

The =o= query parameter is repeatable and maps to
=ListChangesOption=.  Expensive options should be requested only at the
call site that needs them.  Defaults:

- dashboard :: =LABELS=, =DETAILED_ACCOUNTS=, maybe =SUBMIT_REQUIREMENTS=
- review opening :: =ALL_REVISIONS=, =ALL_COMMITS=, =ALL_FILES=,
  =DETAILED_LABELS=, =DETAILED_ACCOUNTS=, =MESSAGES=,
  =REVIEWER_UPDATES=, =DOWNLOAD_COMMANDS=, =CHANGE_ACTIONS=
- selected change details :: prefer =GET /changes/{id}/detail= with
  additional =o= options for revisions/files, instead of re-querying by
  =q=

Avoid deprecated query operators such as =assignee:self=.  Modern Gerrit
still contains compatibility pieces, but we should not build new UI on
those operators.

** Comments and drafts

=ListChangeComments= and =ListChangeDrafts= return a map from file path
to a list of =CommentInfo=.  Published comments have =author= filled;
drafts intentionally do not.  Drafts require an identified user.

=CommentInfo= extends =extensions.client.Comment= and important fields
include:

- =patch_set= :: integer patch set number
- =id= :: URL-encoded comment UUID
- =path= :: file path
- =side= :: absent means revision side; =PARENT= means parent side
- =parent= :: parent number for merge commits, only when side is parent
- =line= :: nil or 0 means file-level comment; normal lines are 1-based
- =range= :: start/end line are 1-based; characters are 0-based; end is
  exclusive
- =in_reply_to= :: parent comment UUID
- =updated= :: Gerrit UTC timestamp string
- =commit_id=, =tag=, =change_message_id=, =unresolved=,
  =fix_suggestions=, =context_lines= may also be present

The data layer must preserve raw comment alists, not only the fields we
initially render.

** Review / draft publishing

=POST /changes/{change-id}/revisions/{revision-id}/review= accepts a
=ReviewInput=.  Field names on the wire are lower-underscore JSON names
(e.g. =work_in_progress=, =omit_duplicate_comments=, =drafts=).  We use
string keys in Emacs JSON alists to avoid symbol/name conversion bugs.

=ReviewInput.drafts= defaults to =KEEP= server-side.  To publish drafts,
post ={"drafts":"PUBLISH_ALL_REVISIONS"}= or ={"drafts":"PUBLISH"}=
to the review endpoint.  This matches egerrit's successful publishing
workflow and avoids inventing a non-standard endpoint.

When posting comments, empty/whitespace-only messages are ignored by
Gerrit (=PostReview.cleanUpComments=).  The UI should prevent empty
comments before sending.

** Binary and text endpoints

Not every endpoint returns JSON:

- =GET /config/server/version= returns a JSON string with the XSSI line.
- =GET /changes/{id}/revisions/{rev}/patch= defaults to base64 text.
  Gerrit 3.14 supports =?raw=, which returns a plain text patch and is
  preferable for Majutsu review buffers.
- =GET /changes/{id}/revisions/{rev}/files/{file-id}/content= returns a
  =BinaryResult= and may be base64-encoded.

Therefore the REST layer has a raw request primitive plus JSON/text
helpers, rather than a single function that always JSON-decodes.

* Dependencies

The new modules require, beyond Majutsu's own modules:

- =cl-lib=, =subr-x=, =seq= (built in)
- =auth-source=, =url-parse=, =url-util= (built in)
- =json= (built in)
- =magit-section= (already a Majutsu dependency)

External binaries: =curl= and =git= must be on =PATH=.

Download resolves a Gerrit change to a local jj change first.  In the
Jujutsu-aware Gerrit model, a change uploaded from this repo normally has
=ChangeInfo.change_id= equal to the raw jj change id from the Git
commit-object =change-id= header.  Download can therefore use that raw
32-character id directly as a jj revset and show the local change in a
diff buffer without fetching anything.

Legacy changes created by older =jj gerrit upload= may instead use a
converted Gerrit key of the form =I<forward-hex-jj-id>6a6a6964=.  Majutsu
keeps a compatibility helper to recover the jj id from that shape, but
this is not the primary model for new changes.

For a change that does not map to a local jj change (someone else's
change, a traditional Git =I...= footer, or a legacy key whose local
change no longer exists), download falls back to fetching the patch set.
Gerrit patch-set refs live under =refs/changes=, which =jj git fetch=
cannot reach, so download runs =git --git-dir $(jj git root) fetch= to
copy the ref into =refs/remotes/<remote>/majutsu/gerrit/...=, then =jj
git import= so jj sees it as a remote bookmark, then =jj new=.

No new entries are added to =Eask=; =auth-source= / =json= are built in,
and =magit-section= is already required.

* Sync/download and cleanup model

Majutsu should expose the same first-class Gerrit lifecycle that we want
from the jj CLI, but with editor-native affordances:

- upload :: call =jj gerrit upload= through the existing transient.
- sync/download :: fetch Gerrit patch sets into jj-visible remote
  bookmarks and optionally create a local child for inspection.
- cleanup :: find Gerrit open changes whose raw jj change id no longer
  appears in the selected local revset, show a dry-run section, and only
  abandon after explicit confirmation.
- review :: use REST DiffInfo/comments/drafts for inline review buffers.

** Download vs sync

=majutsu-gerrit-download-change= is the point-at-change command: it
tries to jump to the local jj change first, and otherwise fetches the
current patch set and creates a child.  This is the interactive analogue
of a future =jj gerrit sync --change <N>=.

A future dashboard-level sync command should not necessarily check out a
child.  It should update remote bookmarks only, for example:

#+begin_example
change/166@gerrit      ; current patch set
change/166/1@gerrit    ; optional exact patch set bookmark
#+end_example

The existing =majutsu-gerrit-download-ref-prefix= (=majutsu/gerrit=)
can remain the internal namespace for fetched refs, but user-facing
sections should render them as Gerrit changes rather than as ordinary
bookmarks.

** Cleanup UI

Majutsu cleanup should be a section buffer, not a silent command.  The
buffer should show:

- the Gerrit query used (=project=, =branch=, =status:open=,
  =owner:self= by default)
- the local revset used as the live set
- every obsolete candidate with number, raw jj id, subject and reason
- actions: abandon selected, abandon all, copy query, open in browser

Default behavior is dry-run.  Abandoning should call either REST
=POST /changes/{id}/abandon= or SSH =gerrit review --abandon=, with a
message that says the jj change id is no longer present in the selected
local revset.

** Relationship to the jj CLI design

The standalone jj design document
=jj-gerrit-sync-cleanup-design.org= specifies the CLI-level algorithm.
Majutsu should share the same identity model and safety rules, but it can
use its existing REST/data/dashboard modules for discovery and display.
In particular:

- Majutsu REST query is better for dashboards and rich detail views.
- Git fetch remains necessary to get commit objects from =refs/changes=.
- Cleanup should reuse dashboard data where available, but must refresh
  before destructive abandon operations.

* Configuration model

The client follows the same "discover from working copy" rule that
=majutsu-gerrit--shared-spec= already uses:

1. =jj config get gerrit.review-url= takes priority.  If set, parse it
   and derive the host, scheme, path prefix and SSL flag.
2. Fall back to the selected jj remote URL (=jj git remote list=) for
   HTTP/HTTPS remotes.
3. SSH remotes are intentionally ignored for REST, because the SSH port
   is the upload port, not the REST port.

Authentication uses =auth-source=, indexed by the resolved
=host[:port]=.  No password ever lives in =defcustom=.

The public REST spec is normalized as:

#+begin_src emacs-lisp
(:host "192.168.110.139:8080"
 :scheme "http"
 :path ""              ; optional reverse-proxy path before /a
 :prefix "/a"
 :user "admin")
#+end_src

The existing =majutsu-gerrit--shared-spec= returns legacy/shared keys
such as =:egerrit-url=, =:gerrit-prefix= and =:ssl=.  M1 adds a small
normalizer that converts that shared spec into the REST spec above.

Configuration entry points (defcustoms):

- =majutsu-gerrit-rest-endpoint-prefix= :: usually =/a= for authenticated
  Gerrit.  Default =/a=.
- =majutsu-gerrit-rest-curl-extra-args= :: list of extra curl flags
  prepended to every request, e.g. =--insecure= for self-signed dev
  servers.

There is no =majutsu-gerrit-host= variable.  All callers either use the
discovered spec from the current Majutsu working copy, or pass an
explicit spec object to lower-level functions.

* REST layer (M1)

** Entry points

The low-level request primitive returns status + raw body and never
assumes JSON:

#+begin_src emacs-lisp
(majutsu-gerrit-rest-request METHOD ENDPOINT &optional DATA SPEC PARAMS HEADERS)
;; => (:status 200 :body "raw response body" :url "...")
#+end_src

JSON and text helpers sit on top:

#+begin_src emacs-lisp
(majutsu-gerrit-rest-json METHOD ENDPOINT &optional DATA SPEC PARAMS)
;; => decoded JSON value, or nil for 204/empty body

(majutsu-gerrit-rest-text METHOD ENDPOINT &optional DATA SPEC PARAMS)
;; => raw text body
#+end_src

- =METHOD= :: =GET= / =POST= / =PUT= / =DELETE=
- =ENDPOINT= :: path component starting with =/=, without query string
- =DATA= :: alist/string that will be JSON-encoded as request body, or nil
- =SPEC= :: optional normalized plist =(:host :scheme :path :prefix :user)=
- =PARAMS= :: alist of query parameters; repeated keys are allowed
- =HEADERS= :: additional request headers

Every endpoint convenience function is a thin wrapper around one of
these helpers.  Async variants are not in M1; they are added when a UI
consumer needs them.

** Curl implementation

Requests are run through =majutsu-process-file=:

#+begin_src emacs-lisp
(majutsu-process-file "curl" nil t nil
                      "--silent" "--show-error" "--location"
                      "--compressed"
                      "--write-out" "\nMAJUTSU_GERRIT_STATUS:%{http_code}\n"
                      ...)
#+end_src

Important details:

- Do not use =--fail= or =--fail-with-body=.  We want curl to return the
  body even for 400/401/409 responses so the client can surface Gerrit's
  plaintext error message.
- JSON endpoints send =Accept: application/json= and =pp=0=.
- JSON request bodies send =Content-Type: application/json; charset=UTF-8=.
- HTTP status is parsed from the explicit marker appended by
  =--write-out=.  For M1 we only support text/JSON responses; ZIP/binary
  download endpoints are out of scope.
- =--location= is allowed; curl does not forward credentials to a
  different host unless explicitly asked to do so.

Reusing =majutsu-process-file= gives us:

- consistent =process-environment= via =majutsu-process-environment=
- TRAMP-aware execution
- consistent UTF-8 coding system handling

** URL and query encoding

Provide helpers:

#+begin_src emacs-lisp
(majutsu-gerrit-rest-encode-id ID)
(majutsu-gerrit-rest-params PARAMS)
(majutsu-gerrit-rest-change-id CHANGE)
#+end_src

Rules:

- Encode every path segment with =url-hexify-string= before inserting it
  into an endpoint.
- Boolean query flags may be represented as =(KEY . t)= and encoded as a
  bare =KEY=, so endpoints like =/patch?raw= can be expressed exactly.
- Repeated options are represented as repeated alist cells, e.g.
  =(("q" . "is:open") ("q" . "owner:self") ("o" . "LABELS"))=.

* Data layer (M2)

=majutsu-gerrit-data.el= defines structs for the Gerrit objects that
Majutsu renders or edits.  Every struct has a =raw= slot holding the
original alist, because Gerrit's REST docs explicitly allow new fields to
appear over time and because later UI work may need fields that M2 does
not yet promote.

Structs:

- =majutsu-gerrit-account= :: raw, account-id, name, email, username,
  display-name
- =majutsu-gerrit-range= :: raw, start-line, start-character, end-line,
  end-character
- =majutsu-gerrit-comment= :: raw, id, patch-set, path, side, parent,
  line, range, in-reply-to, message, updated, author, tag,
  change-message-id, unresolved, commit-id, context-lines,
  source-content-type, fix-suggestions, draft-p
- =majutsu-gerrit-revision= :: raw, id, number, ref, commit, files,
  uploader, real-uploader, fetch, description, current-p
- =majutsu-gerrit-change= :: raw, id, triplet-id, project, branch,
  full-branch, topic, hashtags, change-id, subject, status, created,
  updated, submitted, owner, submitter, current-revision,
  current-revision-number, revisions, labels, reviewers,

#+end_src

Notes:

- For =CommentInfo.side=, absence means revision side.  The constructor
  normalizes this to the symbol =revision=; ="PARENT"= becomes =parent=.
- =range= lines are 1-based, characters are 0-based, and the end
  position is exclusive, matching
  =java/com/google/gerrit/extensions/client/Comment.java=.
- Draft comments do not have =author=.  The constructor takes =DRAFT-P=
  so we can distinguish drafts from published comments reliably.
- =ChangeInfo.id= stays encoded in =raw=; the promoted =id= slot is
  decoded for display.  Prefer =number= for follow-up REST paths.
- Nested maps such as =files=, =labels= and =fetch= remain raw alists in
  M2.  Promote them to dedicated structs only when a UI consumer needs
  typed access.
- The REST layer keeps JSON =false= / =null= as =:json-false= /
  =:json-null= in =raw=; promoted slots normalize both to nil.

* Review rendering (M7)

The review UI should use Gerrit's structured =DiffInfo= endpoint as its
source of truth, not =/patch= text.  This is the main difference from
egerrit and is what lets us match the web UI's review granularity.

Relevant endpoints:

- =GET /changes/{id}/revisions/{rev}/files/{file-id}/diff= returns a
  =DiffInfo= with =content= entries containing =ab=, =a=, =b=,
  =edit_a=, =edit_b=, =skip=, =common= and =due_to_rebase=.
- =GET /changes/{id}/revisions/{rev}/comments= and =/drafts= return
  comments whose =line= / =range= fields use the same side + line
  coordinate system.
- =GET /changes/{id}/revisions/{rev}/ported_comments= and
  =/ported_drafts= ask the server to map older patch set comments onto
  the currently displayed revision.  This should replace egerrit's local
  hunk-guessing for cross-patch-set comment placement.

** Granularity

We can match the web UI for Gerrit's native review model:

| Review target        | Gerrit field(s)                     | Majutsu support |
|----------------------+-------------------------------------+-----------------|
| patch-set level      | path =/PATCHSET_LEVEL=                | yes             |
| file level           | no =line= and no =range=                | yes             |
| side + line          | =side= + =line=                         | yes             |
| multi-line range     | =side= + =range=                        | yes             |
| intra-line selection | =range.start_character/end_character= | yes             |
| parent side          | =side=PARENT= + optional =parent=       | yes             |
| merge parent choice  | =parent= query param / comment field  | later UI polish |
| copied-file porting  | ported comments may duplicate UUIDs | yes, if modeled |

The only caveat is character offsets.  Gerrit and Polygerrit treat
comment character positions as string offsets in the rendered source
line.  Majutsu must compute offsets from the underlying line string, not
from display columns, so tabs, wide characters and line wrapping do not
corrupt ranges.  If we want exact parity for non-BMP Unicode codepoints,
we may need a UTF-16-code-unit conversion helper because the web UI runs
in JavaScript.  For ordinary source code this is usually not visible, but
it should be isolated in one helper rather than spread through the UI.

** Display model

M7 should build a structured review buffer instead of a raw patch buffer.
Every rendered source line gets text properties describing its Gerrit
coordinate:

#+begin_src emacs-lisp
(:path "src/foo.el"
 :side revision              ; or parent
 :parent nil                 ; merge parent number, when applicable
 :line 42                    ; Gerrit line number on that side
 :rangeable t
 :diff-row ROW-ID)
#+end_src

This enables reliable commands:

- point on a line -> create a line comment at that exact Gerrit line
- active region on one side -> create a =CommentRange=
- point on file header -> create a file-level comment
- point on patch-set section -> create a patch-set-level comment
- point on existing thread -> reply to that thread

Display should start unified, not side-by-side:

- It fits Emacs windows better and is easier to make keyboard-native.
- It still preserves side-specific coordinates because each rendered line
  carries =:side= and =:line= properties.
- A side-by-side renderer can be added later using the same intermediate
  row model.

Suggested visual shape:

#+begin_example
,* majutsu-bookmark.el
  [file comment button / existing file-level threads]
@@ change chunk 1 @@
  18 | unchanged context
- 19 | parent-only line
+ 19 | revision-only line      [thread count / unresolved marker]
+ 20 | selected text range highlighted
      └─ thread 85c71774_32968cc4  unresolved
         alice: comment text...
         bob: reply...
#+end_example

Implementation outline:

1. Fetch =DiffInfo= per file, with =intraline= and the selected context.
2. Convert =DiffInfo.content= into an intermediate row list.  Each row
   has optional parent cell, optional revision cell, text, intraline edit
   ranges and Gerrit coordinates.
3. Merge published comments, drafts, ported comments and ported drafts
   into thread objects keyed by stable root comment id + path + side +
   patch set.  Allow duplicate UUIDs from ported copied-file comments.
4. Render rows into a =majutsu-gerrit-review-mode= buffer with text
   properties and overlays.  Do not inherit =diff-mode-map= wholesale.
5. Create comments from text properties rather than by searching hunks in
   a patch buffer.

This removes the class of egerrit bugs around =diff-beginning-of-hunk=:
conversation placement no longer depends on Emacs =diff-mode= navigation
state.  =diff-mode= may still be used for faces, but not for coordinate
reconstruction.

* Open questions deferred to later milestones

- Whether =majutsu-gerrit-review-mode= derives from =majutsu-mode= only,
